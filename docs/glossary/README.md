# Glossário CloudSix Engineering Framework

## Objetivo

Padronizar termos usados no framework para reduzir ambiguidade entre pessoas, agentes de IA e documentos.

## Contexto

Termos como requisito, ADR, playbook, contrato, stack e observabilidade podem variar por equipe. Este glossário define o sentido usado neste repositório.

## Diretrizes

- Use estes termos ao criar documentação nova.
- Quando um projeto tiver vocabulário próprio de domínio, registre glossário local.
- Não use termos genéricos para substituir regras de negócio específicas.

## Termos

| Termo | Definição |
| --- | --- |
| ADR | Registro de decisão arquitetural com contexto, alternativas, decisão e consequências. |
| Agente | Papel especializado executado por pessoa ou IA com missão, escopo e limites definidos. |
| Arquitetura | Organização de responsabilidades, contratos, dados, dependências e operação. |
| Checklist | Lista verificável usada para validar uma entrega ou decisão. |
| Critério de aceite | Condição observável que confirma se uma necessidade foi atendida. |
| Diretriz | Regra prática que orienta decisão ou execução. |
| Escopo | Conjunto de entregas incluídas em uma iniciativa. |
| Fora de escopo | Itens explicitamente não incluídos na entrega atual. |
| Idempotência | Propriedade de uma operação poder ser repetida sem gerar efeito duplicado indevido. |
| Observabilidade | Capacidade de diagnosticar comportamento por logs, métricas, traces, auditoria ou eventos. |
| Playbook | Procedimento operacional para uma situação recorrente. |
| Requisito | Necessidade confirmada de negócio, usuário, operação ou sistema. |
| Risco residual | Risco conhecido que permanece após mitigação ou decisão de aceite. |
| Stack | Conjunto de tecnologias, frameworks, bancos, ferramentas e infraestrutura usados em um projeto. |
| Template | Modelo reutilizável para criar documento ou artefato padronizado. |
| Workflow | Sequência de etapas e responsabilidades para executar um tipo de trabalho. |

## Exemplos

- "Criar ADR" significa documentar decisão relevante, não apenas escolher uma ferramenta.
- "Identificar stack" significa inspecionar o projeto real antes de propor implementação.
- "Critério de aceite" precisa ser testável, não apenas uma intenção.
