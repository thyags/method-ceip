# 11 - Padrões de Documentação

## Objetivo

Padronizar documentação técnica para que seja navegável, acionável e útil para humanos e agentes de IA.

## Contexto

Documentação inconsistente dificulta manutenção e aumenta dependência de pessoas específicas. Este padrão complementa `DOCUMENTATION_STANDARD.md` com regras aplicáveis a documentos de projeto.

## Diretrizes

- Usar português do Brasil.
- Incluir título, objetivo, contexto, diretrizes, exemplos quando aplicável, checklist e conclusão.
- Criar links internos para documentos relacionados.
- Registrar data, status e decisão quando for ADR ou RFC.
- Evitar texto promocional, vago ou sem critério de validação.
- Atualizar índice quando houver novo documento relevante.

## Exemplos

- Um playbook de incidente deve conter papéis, fluxo, comunicação, rollback e pós-incidente.
- Um template de PR deve pedir risco, testes e documentação.
- Um guia de arquitetura deve apontar ADRs relacionados.

## Checklist

- [ ] Estrutura mínima foi usada.
- [ ] Conteúdo é específico ao contexto.
- [ ] Links internos existem quando úteis.
- [ ] Checklist é verificável.
- [ ] Documento não contradiz a constituição.

## Conclusão

Boa documentação reduz ambiguidade e acelera execução com qualidade.
